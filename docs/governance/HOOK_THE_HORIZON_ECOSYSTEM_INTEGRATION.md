# Hook the Horizon — Complete Editorial Ecosystem Integration Contract

**Status:** Active implementation authority  
**Publication:** Hook the Horizon  
**Canonical domain:** https://hookthehorizon.blog  
**Canonical repository:** `racheleliseanderson-ui/hookthehorizon`  
**Portfolio position:** Fifth independent peer publication in the Editorial Ecosystem

## 1. Governing position

Hook the Horizon is a complete publication ecosystem, not an addendum to the other four publications and not a shared-infrastructure substitute. It receives its own inventory, preservation register, evidence matrix, decision register, market analysis, forensic backlog, product roadmap, accessibility review, mobile assessment, release evidence, automation coverage, and living-review cadence.

Shared contracts may be inherited only where they remain brand-neutral. Hook the Horizon retains independent ownership of fishing logic, field authority, visual language, editorial voice, privacy boundaries, conservation controls, application behavior, and reader journeys.

## 2. Publication purpose

Hook the Horizon provides fishing, gear, conservation, regulation, access, condition, and field intelligence from the water ahead. It must help readers interpret conditions, build compatible systems, plan explainable experiments, preserve uncertainty, and record outcomes without promising catches or exposing sensitive locations.

## 3. Core reader jobs

1. Read current conditions and understand what materially changed.
2. Verify regulation, access, closure, and conservation constraints.
3. Build compatible rod, reel, line, leader, lure, and presentation systems.
4. Select bounded presentation experiments from owned inventory.
5. Match plausible hatch or forage classes to evidence-backed patterns.
6. Distinguish verified equipment capability from manufacturer claims or concepts.
7. Preserve private field history, negative outcomes, and inconclusive tests.
8. Export public-safe summaries without exact location disclosure.
9. Use critical functions on mobile and under weak-connectivity conditions.
10. Move between editorial explanation, application use, saved projects, and follow-up field records.

## 5. Evidence hierarchy

Use evidence in this order unless a documented reason requires otherwise:

1. First-party field evidence and verified internal records
2. Official regulations, closures, access records, agency data, weather, hydrology, and conservation sources
3. Manufacturer specifications and direct product documentation
4. Peer-reviewed or professional technical sources
5. Credible market and application evidence
6. Community observations and qualitative signals
7. Unverified claims, which may be retained only as explicitly labeled research leads

Every consequential output must expose source state, geographic scope, freshness, uncertainty, and confidence where relevant.

## 6. Safety and privacy controls

The system must fail closed around:

- exact or sensitive fishing locations
- protected conservation data
- unsupported regulation or access claims
- unsupported safety, medical, legal, or professional claims
- contributor identity or private field routines
- precise location data in exports, logs, analytics, prompts, or application state

Public outputs must use broad geographic context, public-safe descriptions, and recursive location scrubbing. Exact location may remain local-only when the user deliberately stores it.

## 7. Repository and release boundaries

Release authority remains:

- `wordpress-theme/`
- `wordpress-plugin/hook-content/`
- governed application source under `applications/`

Legacy paths remain preserved until reconciliation is complete. Source merge does not publish or activate WordPress. Production progression requires exact artifact parity, rollback evidence, critical-route validation, accessibility checks, monitoring, and absence of an A3 exception.

## 8. Research and development workflow

### Intake

Every new concept, source, user signal, field observation, issue, or application change enters through a structured intake record.

Required fields:

- item ID
- publication
- reader job
- source system
- authority state
- evidence class
- evidence confidence
- location sensitivity
- preservation requirement
- development decision
- priority
- owner
- next action
- review date
- trigger

### Triage

Use only the approved decisions:

- Build Now
- Refine Existing
- Expand
- Reposition
- Repackage
- Connect
- Combine Carefully
- Separate and Clarify
- Strengthen the Foundation
- Research Further
- Incubate
- Defer Development
- Preserve for Future Enhancement

Use priorities P0, P1, P2, P3, Incubation, Deferred, and Future Enhancement.

### Evidence review

For each active item:

1. verify provenance and freshness;
2. classify official, firsthand, scholarly, manufacturer, market, or qualitative evidence;
3. record applicability and limitations;
4. check contradiction and correction status;
5. identify coverage, method, and decision gaps;
6. map the evidence to a reader-facing claim or implementation decision.

### Build and validation

1. create or update a bounded issue;
2. preserve current assets and source history;
3. implement on a branch;
4. run proportional tests;
5. validate privacy, sensitive-location, evidence, accessibility, and rollback controls;
6. generate release evidence;
7. merge when checks pass and no A3 exception exists;
8. verify runtime parity separately.

### Review cadence

- Monthly: regulations, platform capability, application competition, product data, market signals, and active P0/P1 evidence.
- Quarterly: full inventory, preservation state, decisions, accessibility, technical dependencies, reader feedback, and monetization signals.
- Immediate: material regulation, closure, safety, conservation, privacy, platform, evaluator, accessibility, or contradictory-user-behavior changes.

## 10. Required automation coverage

Hook the Horizon must be included in every portfolio workflow that handles publication membership, research intake, evidence review, article routing, image coverage, metadata validation, accessibility, release parity, opportunity-to-evidence conversion, scheduled review, and exception routing.

Minimum automation events:

- repository change detected
- new or changed Drive authority detected
- Notion work item created or moved
- evidence review due
- source freshness expired
- regulation or closure source changed
- metadata validation failed
- sensitive-location check failed
- accessibility or CI check failed
- release artifact produced
- runtime parity verified or failed
- quarterly inventory review due

- runtime parity is recorded independently from source merge;
- no four-publication language remains active authority.
